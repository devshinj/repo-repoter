# HRMS 업무 등록 전 자동 동기화 설계

## 개요

HRMS 업무 등록 시 캐시 DB의 커밋 데이터가 최신이 아닐 수 있다. 등록 직전에 해당 매핑의 저장소들을 자동 동기화하여 커밋 누락을 방지한다.

## 배경

- 현재 `POST /api/hrms/register`는 SQLite 캐시에서 커밋을 조회한다
- 스케줄러 자동 동기화(15분 간격)가 실패하거나 오래된 경우 커밋이 누락될 수 있다
- 사용자가 수동으로 저장소 동기화 후 업무 등록하는 것은 번거롭다

## 설계

### 서버 변경 — `POST /api/hrms/register`

기존 흐름에 동기화 단계를 삽입한다:

```
인증 → 매핑 조회 → 중복 체크 → [동기화] → 캐시 커밋 조회 → Gemini 생성 → HRMS 등록
```

#### 동기화 단계 상세

1. 매핑에 연결된 저장소 목록(`mapping.repos`)을 순회
2. 각 저장소에 `syncOneRepo(db, userId, repo)` 호출
3. `syncOneRepo`가 `null` 반환(이미 syncing 상태) 시 → 에러로 처리, 즉시 중단
4. `syncOneRepo`에서 예외 발생 시 → 에러로 처리, 즉시 중단
5. 모든 저장소 동기화 성공 후 기존 커밋 조회 로직 진행

#### 실패 정책

- **하나라도 실패하면 전체 중단** — 부분 성공 허용하지 않음
- 실패 시 응답: `{ error: "동기화 실패", failedRepo: "owner/repo", detail: "..." }`
- 동기화 실패 이력은 기존 `sync_log` 테이블에 자동 기록됨

#### 응답 구조 변경

성공 응답에 동기화 결과를 포함한다:

```json
{
  "message": "Task registered",
  "hrmsTaskId": 123,
  "title": "...",
  "estimatedMinutes": 120,
  "action": "created",
  "syncResults": [
    { "repo": "owner/repo1", "commitsProcessed": 5 },
    { "repo": "owner/repo2", "commitsProcessed": 0 }
  ]
}
```

### 클라이언트 변경 — 단계별 프로그레스

등록 버튼 클릭 후 시간 기반 시뮬레이션으로 진행 단계를 표시한다:

```
● 저장소 동기화 중... (1/3)
● 저장소 동기화 중... (2/3)
● 저장소 동기화 중... (3/3)
● 업무 내용 생성 중...
● HRMS 등록 중...
✓ 등록 완료
```

#### 시뮬레이션 로직

- 저장소 수(`mapping.repos.length`)를 기반으로 단계 목록 생성
- 각 동기화 단계: 2초 간격으로 전환
- "업무 내용 생성 중": 동기화 단계 종료 후 전환
- "HRMS 등록 중": 전체 시간의 마지막 구간
- API 응답 수신 시 즉시 "등록 완료" 또는 에러 표시

#### 에러 표시

동기화 실패 시:
```
✗ 저장소 동기화 실패 (owner/repo2)
  "에러 메시지..."
```

### 기존 중복 방어와의 호환

- `trySyncStart()` CAS 잠금은 그대로 유지
- 스케줄러 자동 동기화와 동시 실행 시: 해당 저장소가 이미 syncing이면 register API에서 에러 반환
- 이 경우 사용자에게 "동기화가 진행 중입니다. 잠시 후 다시 시도해주세요" 안내

## 변경 파일

| 파일 | 변경 내용 |
|------|-----------|
| `src/app/api/hrms/register/route.ts` | 동기화 단계 삽입, 응답에 syncResults 추가 |
| HRMS 등록 UI 컴포넌트 | 프로그레스 시뮬레이션 추가 |

## 범위 밖

- SSE/WebSocket 실시간 스트리밍 — 저장소 수가 적어 시뮬레이션으로 충분
- 동기화 스킵 옵션 — 불필요한 복잡도 추가
